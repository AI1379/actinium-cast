# Architecture

## 1. 系统概述

Actinium Cast 是一个基于 BitTorrent DHT 网络与轻量级区块链构建的纯匿名、抗审查、防垃圾信息的内容分发平台。系统采用“重客户端密码学 + P2P 数据网络 + 区块链仲裁”的三层架构，并提供友好的 Web 网关以降低普通用户的使用门槛。

## 2. 核心架构设计

### 2.1 存储与网络层 (P2P Layer)

* **底层协议**：利用现有的 BitTorrent Mainline DHT 网络，利用 **BEP 44** 协议（Mutable Data）存储短文本、评论和交互指令。
* **网关节点 (Relays)**：任何人均可部署后端的 Web 服务器作为 P2P 网络的节点。它们负责从 DHT 网络拉取数据、缓存、并提供 HTTP API 供前端读取。网关**不保存**用户的任何身份凭证或私钥。
* **网关发现与网状同步 (Mesh Syncing)**：为解决纯 DHT 中无缓存节点无法自动发现数据的问题，网关利用 **BEP 5** (Tracker / `announce_peer`) 协议宣告自身并发现其他存活的同伴网关。网关间通过 HTTP 点对点拉取“已知公钥录”，随后再向 DHT 提取数据本身，以此构建无需中心服务器、兼顾隐私的抗清空恢复能力。
* **NAT穿透与个人化节点 (NAT Traversal - 远期)**：在当前架构下网关通常需带有公网 IP的云服务器做支撑。为了实现极致去中心化，使普通家用网络下的个人电脑也能作为全尺寸网关参与 Mesh，未来架构计划在底层引入完整的类 BT 穿透外挂，包括 UPnP / NAT-PMP 自动端口映射及基于 UDP (uTP 等效) 的防火墙双向打洞 (Hole Punching)。

### 2.2 共识仲裁层 (Blockchain Layer)

* **轻量级区块链**：采用低手续费的 EVM 兼容链（如 Polygon）或基于 Substrate 定制的应用链。
* **职责 (治理)**：绝对不存储业务内容，仅运行一个**管理员阈值多签智能合约**。用于记录合法的管理员公钥集合，以及广播最终的“全局屏蔽指令”。
* **职责 (可选备份)**：支持特定“归档网关 (Archive Gateway)”周期性通过触发**极低成本的事件日志 (Event Logs)**，将时间周期内积累的公钥名册全量备份上链。为该项目提供能够抵御全网断电下线的“末日冷启动灾难恢复方案”。

### 2.3 客户端与执行层 (Client/WASM Layer)

* **执行环境**：普通的 Web 浏览器，核心逻辑由 Rust 编译的 WebAssembly (Wasm) 模块驱动。
* **本地客户端**：基于 Rust 实现，可以利用 Tauri，也可以只是一个 TUI 应用，用于初步测试，后期迁移到 wasm。
* **职责**：本地随机生成 Ed25519 密钥对、计算防垃圾 PoW、对数据进行签名、打包零知识证明数据。

## 3. 核心机制实现

### 3.1 匿名身份与普通操作

* **发帖/评论**：用户前端在本地生成 Ed25519 私钥。将帖子内容哈希后，使用私钥签名，连同公钥推送到网关，网关广播至 DHT。
* **自行删帖**：利用 BEP 44 协议的 `Sequence Number` 机制，用户用同一私钥对一个空内容进行签名并提升序列号，覆盖原有的 DHT 节点数据。前端检测到新序列号即判定为内容已删除。
* **点赞/点踩**：构造成一种特殊的引用类 DHT 消息（类似 Nostr 协议），结构为：`[目标帖子Hash, 操作类型, PoW证明, 用户签名]`。

### 3.2 防垃圾冲击 (Anti-Spam)

* **纯客户端 Hashcash**：没有任何中心化风控。每次发帖、点赞或评论，前端 Wasm 模块必须利用 CPU 满载计算出一个符合特定难度（如前 N 位为 0）的 SHA-256 哈希值。
* **网络拦截**：网关节点和客户端在拉取到 DHT 数据后，第一时间验证 PoW。若 PoW 难度不达标，直接丢弃该数据，防止恶意脚本瞬间发送百万条垃圾信息。

### 3.3 管理员动态多签与防黑化机制

* **动态身份与初始化**：
  * **代码解耦**：智能合约的源代码（Rust/Solidity）中没有任何具体的公钥信息。
  * **初始注入**：在系统首次上线（部署智能合约）时，将初始的管理员公钥集合（例如 7 个公钥）作为**初始化参数（Constructor Arguments）**动态注入到合约状态中。

* **管理员席位轮换 (Admin Rotation)**：
  * 智能合约内置 `变更管理员` 的逻辑接口。
  * 当需要更换管理员（例如老成员离职、密钥疑似泄露）时，由现有管理员发起一个“席位变更提议”：`[操作: 替换, 撤销公钥: A, 新增公钥: X]`。
  * 该提议同样需要满足阈值（如 4/7 多签）才能在链上生效。生效后，合约内部的合法管理员名单自动更新，后续的屏蔽指令将基于新名单进行校验。
  * *(兼容线下交接：由于系统不过分强求管理员的安全隔离，老管理员在离任时，也可以直接在物理世界将私钥 U 盘移交给继任者，无需触发链上轮换逻辑。)*

* **全局屏蔽指令流转**：
  * 管理员发现恶意内容后，在 P2P 网络中广播“屏蔽提议”。
  * 当集齐阈值数量（如 4 个）的当前合法管理员签名后，该提议被提交至区块链智能合约。
  * 合约验证签名者都在**当前最新**的管理员名册内，验证无误后，在链上抛出 `TombstoneEvent(PostHash)`。所有网关节点监听到该事件，自动在本地缓存中过滤掉对应哈希的内容。

## 4. 关键工作流示例：普通用户通过 Web 网关发帖

1. 用户访问 `gateway.example.com`，下载 HTML 与 Wasm 核心库。
2. 用户在文本框输入内容并点击发送。
3. Wasm 在浏览器后台：
    * 随机生成一次性或持久化的密钥对。
    * 对内容和时间戳执行 PoW 运算（耗时约 3~5 秒）。
    * 用私钥对内容 + PoW 结果进行 Ed25519 签名。
4. 前端将打包好的密文包通过 HTTPS 发给 Web 网关。
5. Web 网关验证签名和 PoW，验证通过后，按照 BEP 44 协议封装，注入全球 BitTorrent DHT 网络。
6. 其他网关通过 DHT 同步到该数据，全球用户即可在各自的 Web 界面上看到该贴。
